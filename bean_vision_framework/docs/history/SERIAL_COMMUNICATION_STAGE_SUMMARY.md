# 视觉-电控串口通信稳定性阶段性设计评审文档

## 1. 评审目的

目前视觉端（NUC）与电控端（STM32）已完成基础串口通信联调。为规避盲目推进代码修改的风险，现对通信稳定性及架构设计进行阶段性整理，提请学长对以下核心问题进行评审与指导：

1. **进度与方向**：当前稳定性设计方向及推进顺序是否合理？
2. **机制必要性**：`HELLO / READY` 握手与 `HEARTBEAT` 心跳机制是否有必要在此阶段引入？
3. **安全策略**：`WARNING` 状态的业务冻结与断线不续跑策略是否合理？
4. **架构分层**：新增 `comm_state.c` 模块属于合理分层还是过度设计？

---

## 2. 当前进展与暴露风险

### 2.1 已完成的基础

* **物理与协议层**：USB CDC 基础通信链路已通；NUC 侧完成串口打开与收发；STM32 侧完成 `protocol.c` 组帧与解帧。
* **校验与机制**：实现 CRC16-MODBUS 校验及基础 ACK 响应机制。
* **业务联调**：`ARRIVE_BEAN`（数据到达）与 `BEAN_BIND`（数据返回）基础业务链路已打通。

### 2.2 暴露的主要风险

* **启动不同步**：NUC 与 STM32 启动顺序不固定。若 NUC 先于 STM32 USB CDC 初始化完成前发送业务帧，会导致数据丢失及通信误判。
* **掉线感知滞后**：运行中若发生物理线松动、NUC 崩溃或 MCU 复位，电控端无法区分“视觉暂时无数据”与“视觉已失联”。
* **状态不一致**：通信异常复位后，两端可能处于不同的业务状态机节点（例如 STM32 在等待下一步指令，而 NUC 重启后状态已重置），直接续跑会导致任务错位。
* **诊断手段缺失**：当前 CRC 错误采用静默丢弃，缺乏统计维度，链路噪声与协议解析错误无法定位。

---

## 3. 通信功能架构分层方案

为保证代码的可迭代性，建议将通信逻辑划分为三个耦合度低的层次：

```
+-------------------------------------------------------------+
|                        业务层 (Business)                    |
|       - 关心 comm_is_ready() 状态                           |
|       - 处理具体业务命令 (e.g., ARIVE_BEAN)                  |
+-------------------------------------------------------------+
                              |
+-------------------------------------------------------------+
|                      通信状态层 (comm_state)                |
|       - 管理链路生命周期 (OFF / WAIT_HELLO / READY / WARNING) |
|       - 维护心跳与超时检测、维护 crc_error_count             |
+-------------------------------------------------------------+
                              |
+-------------------------------------------------------------+
|                        底层协议层 (protocol)                 |
|       - 仅负责纯粹的帧结构打包、解包与 CRC16 校验           |
+-------------------------------------------------------------+

```

### 3.1 各层职责边界

* **底层协议层 (`protocol.c`)**：只负责帧头、CMD、LENGTH、SEQ、PAYLOAD、CRC16 的组帧与解帧，保持底层纯净，不包含任何状态机与超时逻辑。
* **通信状态层 (`comm_state.c` 新增)**：轻量级链路健康管理模块。维护通信状态机，负责心跳收发、超时判定及提供 `comm_is_ready()` 接口。
* **业务层**：仅通过 `comm_is_ready()` 查询通信是否可用，负责具体业务逻辑分发。

## 4. 链路管理与控制命令设计

### 4.1 握手与心跳机制

* **握手 (HELLO / READY)**：解决启动顺序不确定性。由作为 Host 的 NUC 主动发起 `HELLO`，作为 Device 的 STM32 被动响应 `READY`，完成建联。
* **心跳 (HEARTBEAT / HB_ACK)**：解决运行期保活问题。NUC 定期发送心跳，STM32 响应，用以区分“设备节点存在”与“对端程序正常运行”。

### 4.2 通信控制命令集

| CMD | 指令值 | 方向 | Payload 含义 | 详细说明 |
| --- | --- | --- | --- | --- |
| `HELLO` | `0x15` | NUC $\rightarrow$ STM32 | `[version, reserved]` | NUC 发起握手请求 |
| `READY` | `0x16` | STM32 $\rightarrow$ NUC | `[version, reserved]` | STM32 确认链路建立 |
| `HEARTBEAT` | `0x17` | NUC $\rightarrow$ STM32 | 无 | NUC 周期心跳包 |
| `HB_ACK` | `0x18` | STM32 $\rightarrow$ NUC | 无 | STM32 心跳响应确认 |

### 4.3 状态机时序参数

* **HELLO 重试间隔**：`200ms`（NUC 未收到 READY 前的发送频率）
* **心跳发送间隔**：`200ms`（链路 READY 后 NUC 的发送频率）
* **WARNING 判定阈值**：`500ms`（STM32 超过此时间未收到任何有效帧则触发）
* **OFF 判定阈值**：`3000ms`（长时间无通信，判定为彻底断开）
* **诊断数据**：两端维护 `crc_error_count`，用于现场调测诊断。

## 5. 异常处理与边界定义

### 5.1 `READY` 状态的语义边界

> **核心共识**：`READY` 仅代表**通信层链路就绪**。

* **包含**：USB CDC 枚举成功、协议正常组解帧、`comm_state` 握手成功。
* **不包含**：电机就绪、夹爪复位、整车模式允许执行。后续业务执行需由联合条件决定（例如：`comm_is_ready() && motor_is_ready()`）。

### 5.2 `WARNING` 状态安全策略

当触发 `WARNING`（超时未收到数据）时，系统进入安全挂起状态，而非整体崩溃：

* **允许继续的动作（电控本地闭环）**：底盘/步进安全停机、夹爪保持当前位置（防止掉豆）、继续等待通信恢复。
* **必须冻结的动作（视觉协作相关）**：禁止使用可能过期的视觉数据、暂停后续协作任务、禁止自动进入下一步骤。

### 5.3 断线恢复策略

本阶段不设计复杂的“断点续跑”与双端 `task_id / step_id` 同步。

* **策略**：**安全暂停 + 重新建联**。
* **流程**：通信断开 $\rightarrow$ 电控进入 `WARNING` 并冻结动作 $\rightarrow$ 通信恢复后允许重新进行 `HELLO / READY` 握手 $\rightarrow$ 业务层不自动续跑，而是**挂起并等待上层/人工重新触发业务确认**。

## 6. 职责划分

### 6.1 视觉端 (NUC) 职责

* 决定 `HELLO` 握手与 `HEARTBEAT` 心跳的发送时机与重连策略。
* 负责视觉识别算法及业务 Payload 数据的准确性。

### 6.2 电控端 (STM32) 职责

* 负责 `WARNING` 超时判定、本地安全停机与电机控制逻辑。
* 负责 USB CDC 接收缓冲管理、多帧解析顺序以及 NRF 遥控链路安全覆盖。

### 6.3 双方共同冻结接口

* 物理波特率、帧头/帧尾格式、CRC16 校验标准、CMD 编号及 Payload 结构体定义。

## 8. 提请学长重点指导问题

1. 在基础通信已通的前提下，现阶段引入 `HELLO / READY` 与 `HEARTBEAT` 机制是否会**过度设计**？还是说对实车联调是**必要的刚需**？
2. `WARNING` 状态下“**电控本地闭环继续，视觉协作任务冻结**”以及“**重连后不自动续跑**”的策略是否符合比赛实车调测习惯？
3. 参数设置（心跳 `200ms` / 报警 `500ms` / 断开 `3000ms`）是否合理，是否需要根据控制周期进一步收紧或放宽？
4. 新增轻量级 `comm_state.c` 进行解耦的方向是否正确？是否存在不符合实际工程经验的隐患？
